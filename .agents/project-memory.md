# 工资条 PWA 项目记忆 Agent

## 角色

这是本项目的长期记忆文件。后续继续开发 `工资条PWA` 时，先阅读本文件，再修改规则、页面或数据结构。

## 规则优先级

工资计算、请假解析、工龄加薪、春节特殊结算等所有算法规则，以 `工资条PWA/.agents/payroll-rules-draft.md` 为最高优先级。

本文件只记录项目背景、技术方案、当前功能和开发原则。若本文件中的历史描述与 `payroll-rules-draft.md` 冲突，一律以 `payroll-rules-draft.md` 为准。

`工资条PWA/.agents/payroll-rules-draft.pdf` 是用户自己的备份文件，不作为开发依据，也不要主动修改或提交。

## 项目目标

做一个老板自己手机上使用的工资条生成工具：

- 不先上传 GitHub，先把本地网页做好。
- 后续计划部署到 GitHub Pages。
- 形式是静态 PWA 网页应用，可添加到手机桌面。
- 不依赖后端服务器，默认零成本使用。
- 每月粘贴简单请假记录，自动生成每个工人的工资条。
- 工资条要能拆开，分别复制后发给不同工人。

## 当前技术方案

- 纯静态文件：`index.html`、`styles.css`、`app.js`
- 云端数据：Supabase Postgres + Realtime
- 当前访问方式：免登录单人私用版，打开链接直接读写同一份数据
- 安全策略：所有业务表仍启用 RLS，但策略允许匿名读写；安全依赖链接不外传
- PWA：`manifest.webmanifest` + `sw.js`
- 部署目标：GitHub Pages，仓库名暂定 `wage-slip-pwa`
- 无构建工具，无 npm 依赖，便于直接托管和维护

## Excel 导入版

- 独立页面：`excel.html`
- 独立脚本：`excel-app.js`
- 模板文件：`templates/工人资料模板.xlsx`
- 模板生成脚本：`create_excel_template.py`
- Excel 版不连接 Supabase，工人资料来自上传的 Excel。
- 上传 Excel 后，网页会把识别出的最新版工人数据保存到当前浏览器 `localStorage`，下次打开 Excel 版不用重复上传。
- Excel 版与数据库版互相独立，互不影响。
- Excel 模板固定 4 张表：
  - `工人资料`
  - `中断记录`
  - `规则参数`
  - `填写说明`
- 不要随意修改 Excel 工作表名称和第一行字段名，否则网页可能无法识别。

## 当前功能

- 工人档案：姓名、基本工资、职位津贴、满年日期、备注、在职状态
- 月度录入：粘贴自然格式请假记录，月份由老板写在记录里，例如第一行 `6月` 或 `2026年6月`
- 自动解析：整天请假、半天请假、预支金额
- 工资计算：已开始按 `payroll-rules-draft.md` 实现，后续继续以该文件为准
- 工资条输出：
  - 每个工人独立工资条卡片
  - 每个卡片有独立复制按钮
  - 生成顺序优先按老板粘贴记录里的工人顺序，不按数据库排序强行重排
- 当前主流程不强调月度保存和历史查询；老板主要需要当月快速生成、逐个复制发送
- 备份：导出 JSON；云端数据保存在 Supabase

## 2026-07-23 收尾维护记录

当前线上地址：

- GitHub Pages 主地址：`https://cby172.github.io/wage-slip-pwa/`
- 默认主界面是数据库版，顶部可切换 Excel 版。
- Excel 独立页仍可直接打开：`https://cby172.github.io/wage-slip-pwa/excel.html`

当前版本形态：

- 数据库版：`index.html` + `app.js` + Supabase。
- Excel 版：`excel.html` + `excel-app.js`，使用上传 Excel 后保存到浏览器本地 `localStorage` 的方式。
- 两个版本应尽量保持主流程一致：`算工资` 是第一入口，老板粘贴请假记录后点 `生成工资条`。
- Excel 版和数据库版互相独立，互不影响。不要为了省代码把两边数据混在一起。

当前老板确认过的 UI 口径：

- `年份兜底` 不显示给老板，代码内部自动使用当前年份；如果输入文本里写了 `2026年6月`，仍优先用文本年份。
- `请假记录` 输入框默认为空，只显示浅黑色示例占位。
- 输入框没有点击时显示占位示例；点击后占位示例消失，只剩光标，便于老板直接粘贴。
- `示例数据` 是按钮，点击后才把示例填入输入框。
- `生成工资条` 后，每个工人的工资条仍单独显示，每个单人工资条有 `复制` 按钮。
- 底部 `工资汇总` 不是表格汇总，不是“应发/实发/预支”列表；它的含义是把每个工人的工资条原文按顺序拼接，中间空一行。
- `工资汇总` 右上角按钮文案只写 `复制`，不要写 `复制汇总`。

当前统计面板口径：

- 数据库版和 Excel 版都有最后一个 `统计` 栏目。
- 统计只读，不可在统计面板里改数据。
- 统计默认按工龄从大到小排序。
- 前两项必须是 `姓名` 和 `工龄`。
- 工龄用红色突出，单位是年，保留 2 位小数。
- `津贴` 和 `工资合计` 必须分开，不要写成 `津贴/合计`，避免老板误会。
- 手机端统计表可以横向滑动，尤其 Excel 版嵌入主页面时不要让表格撑破页面。

当前数据库版工人状态联动口径：

- `workers.active` 和 `workers.interruptions` 是强关联数据，不能只改其中一个。
- 点击 `停用` 时，系统必须把工人改为不在职，并自动在 `interruptions` 里新增一条 `{ start: 今日日期, end: "", note: "停用" }`；如果已经有未结束中断记录，不要重复新增。
- 点击 `启用` 时，系统必须把工人改为在职，并自动把最新一条未结束的中断记录 `end` 补为今日日期。
- 修改工人表单里如果直接改 `在职` 状态，也要执行同样的联动，避免数据库状态和工龄统计脱节。
- `interruptions` 直接影响工龄、闲置时间、底薪自动计算公式和工资生成参与状态；未结束中断记录按当前日期持续扣除工龄。

已踩坑和后续必须避免：

- 不要把老板要的 `工资汇总` 理解成结构化统计列表；它就是拼接后的工资条原文。
- 不要把 `复制` 按钮文案写成长文案，老板更喜欢短按钮。
- 不要把请假示例直接填进输入框；那会影响老板粘贴。示例应该是占位，或通过 `示例数据` 按钮主动填入。
- 不要让 Excel 版统计表撑出手机屏幕；宽表格必须在内部横向滚动。
- 不要重新加入登录流程。用户已经明确要求免登录、打开链接直接用。
- 不要把 GitHub token、Supabase service role key 或任何私密密钥写进文件；只能在临时命令中使用发布 token。
- 用户曾在聊天中发过 GitHub token，后续维护时应提醒用户在 GitHub 中撤销旧 token，并换用更安全的部署方式。
- 部署到 GitHub Pages 后，手机端经常会被 `sw.js` 和浏览器缓存影响；每次改 `app.js`、`excel-app.js`、`styles.css`、`index.html`、`excel.html` 后，都要更新引用版本号和 `sw.js` 的 `CACHE_NAME`。
- GitHub Pages 有 CDN 延迟，API 更新成功后公开地址可能几十秒后才看到新文件；验证时用带 `?fresh=...` 的 URL 拉取。
- 本地 Git 可能有 `dubious ownership` 问题，不要为此乱改全局配置；可以使用 GitHub API 逐文件更新，或显式 `-c safe.directory=...`。

## 开发原则

- 优先保持手机端简单可用，不做复杂后台。
- 写算法前必须先阅读 `工资条PWA/.agents/payroll-rules-draft.md`。
- 规则变化集中放在 `app.js` 的解析和计算函数里；必要时再拆分规则模块。
- 修改计算前先根据 `payroll-rules-draft.md` 补规则样例校验。
- 不删除离职工人，使用停用状态。
- 备份格式要保留 `schemaVersion`，以后改数据结构需要做兼容迁移。
- 部署前记得更新 `sw.js` 缓存版本，避免旧缓存影响手机端。
- 前端只能使用 Supabase `anon public key`，不能把 `service_role key` 写进任何前端文件。
- 用户明确要求取消登录；除非用户重新要求，不要恢复邮箱/密码登录。
